================================================================================
COMPREHENSIVE NOTIFICATION RULES BY PAGE/FEATURE
================================================================================

This document maps every function/action in each page to its notification rules.
Follow this guide when implementing notifications throughout the application.

================================================================================
1. USER MANAGEMENT (SalesAgentsPage, SystemAdminPage)
================================================================================

PAGE: SalesAgentsPage
FUNCTIONS:
----------

1. CREATE USER
   Function: handleCreateAgent / createUserProfile
   Notification Type: user_created
   Recipients: 
     - super_admin (if exists)
     - admin (if exists)
     - The newly created user (welcome notification)
   Message Format:
     - To super_admin/admin: "{creator_name} has created a new user: {new_user_name} ({new_user_role})"
     - To new user: "Welcome! Your account has been created. Email: {email}, Temporary Password: {temp_password}"

2. UPDATE USER
   Function: handleUpdateAgent
   Notification Type: user_updated
   Recipients:
     - super_admin (if exists)
     - admin (if exists)
     - The updated user (if their own data changed)
   Message Format:
     - To super_admin/admin: "{updater_name} has updated user: {user_name}"
     - To updated user: "Your profile has been updated by {updater_name}"

3. DELETE USER
   Function: handleDeleteAgent
   Notification Type: user_deleted
   Recipients:
     - super_admin (if exists)
     - admin (if exists)
   Message Format:
     - "{deleter_name} has deleted user: {deleted_user_name} ({deleted_user_role})"

4. RESET PASSWORD
   Function: handleResetPassword
   Notification Type: password_reset
   Recipients:
     - The user whose password was reset
   Message Format:
     - "Your password has been reset by {resetter_name}. New temporary password: {temp_password}"

5. CHANGE USER STATUS (Active/Inactive)
   Function: handleStatusChange
   Notification Type: user_status_changed
   Recipients:
     - super_admin (if exists)
     - admin (if exists)
     - The user whose status changed
   Message Format:
     - To super_admin/admin: "{changer_name} has changed {user_name}'s status to {new_status}"
     - To user: "Your account status has been changed to {new_status} by {changer_name}"

6. ALLOCATE STOCK TO AGENT (from SalesAgentsPage)
   Function: allocate_to_agent RPC
   Notification Type: stock_allocated_to_agent
   Recipients:
     - The agent receiving stock
     - super_admin (if allocation involves manager/team_leader)
     - admin (if allocation involves manager/team_leader)
   Message Format:
     - To agent: "You have received {quantity} units of {brand_name} - {variant_name} from {allocator_name}"
     - To super_admin/admin: "{allocator_name} has allocated {quantity} units of {brand_name} - {variant_name} to {agent_name}"

================================================================================
NOTIFICATION PRIORITY LEVELS:
================================================================================

CRITICAL (Immediate, High Visibility):
- Order approvals/rejections
- Stock allocations
- Inventory request approvals/denials
- Purchase order approvals/rejections
- User creation/deletion
- Company creation/deletion

HIGH (Important, Standard Notification):
- New orders created
- Inventory requests created
- Client approvals/rejections
- Task assignments
- Team member assignments/unassignments

MEDIUM (Informational):
- Profile updates
- Order updates
- Task updates/completions
- Inventory updates
- Client updates

LOW (Background, Optional):
- Avatar uploads
- Password changes
- Profile updates (self)
- Calendar events

================================================================================
NOTIFICATION GROUPING RULES:
================================================================================

1. BULK OPERATIONS:
   - If multiple users are created at once, send individual notifications per user
   - If stock is allocated to multiple recipients, send individual notifications
   - Group similar notifications if they occur within 5 seconds

2. CASCADE NOTIFICATIONS:
   - When an action triggers multiple notifications, send them in order:
     1. Direct recipient (immediate)
     2. Supervisors/managers (immediate)
     3. System admins (immediate)
     4. Confirmation to actor (immediate)

3. DUPLICATE PREVENTION:
   - Don't send notification if user performs action on themselves (except confirmations)
   - Don't send notification if action doesn't change state
   - Batch notifications that occur within 1 second

================================================================================
IMPLEMENTATION NOTES:
================================================================================

1. All notifications should include:
   - Timestamp
   - Action performer name
