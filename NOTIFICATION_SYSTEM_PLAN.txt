================================================================================
NOTIFICATION SYSTEM PLAN
================================================================================

OVERVIEW:
This document outlines the notification system rules, message formats, and 
recipients for different events in the MultiTenantB2B system.

================================================================================
USER ROLES HIERARCHY:
================================================================================
1. super_admin (User 1) - Highest level, sees all system events
2. admin (User 2) - Administrative level, sees system events
3. manager (User 3) - Management level
4. team_leader (User 4) - Team management level
5. mobile_sales (User 5) - Sales agent level

================================================================================
NOTIFICATION RULES:
================================================================================

RULE 1: SYSTEM-WIDE EVENTS
---------------------------
Events that affect the entire system should notify super_admin AND admin.

Examples:
- User creation/deletion
- User role changes
- System configuration changes
- Company-wide announcements

Recipients: super_admin, admin

Message Format Examples:
- "{creator_name} has created a new user: {new_user_name}"
- "{creator_name} has deleted user: {deleted_user_name}"
- "{creator_name} has changed {user_name}'s role from {old_role} to {new_role}"


RULE 2: STOCK ALLOCATIONS
-------------------------
Stock allocations create notifications for relevant parties based on the 
allocation path.

Allocation Paths:
1. super_admin → manager
2. super_admin → team_leader
3. admin → manager
4. admin → team_leader
5. manager → team_leader
6. team_leader → mobile_sales

Recipients:
- The recipient of the allocation (who receives the stock)
- The sender (who allocated the stock) - optional, for confirmation
- super_admin (if allocation involves manager or team_leader)
- admin (if allocation involves manager or team_leader)

Message Format:
"{sender_name} has allocated {quantity} units of {brand_name} - {variant_name} to {recipient_name}"

Example:
"John Doe has allocated 50 units of Brand A - Product X to Jane Smith"


RULE 3: INVENTORY REQUESTS
--------------------------
Inventory requests follow the chain: mobile_sales → team_leader → admin

Notifications sent to:
- Next approver in the chain
- Original requester (when request is approved/denied)

Message Formats:
- Request created: "{requester_name} has requested {quantity} units of {product_name}"
- Request approved: "Your request for {quantity} units of {product_name} has been approved by {approver_name}"
- Request denied: "Your request for {quantity} units of {product_name} has been denied by {approver_name}. Reason: {reason}"


RULE 4: CLIENT ORDERS
---------------------
Client order notifications follow similar chain as inventory requests.

Notifications sent to:
- Next approver (team_leader or admin)
- Order creator (when order is approved/denied)

Message Formats:
- Order created: "{agent_name} has created a new order: {order_number} for {client_name}"
- Order approved: "Order {order_number} has been approved by {approver_name}"
- Order denied: "Order {order_number} has been denied by {approver_name}. Reason: {reason}"


RULE 5: PURCHASE ORDERS
------------------------
Purchase orders are typically created by team_leader or manager and approved by admin.

Notifications sent to:
- Admin (when PO is created)
- Creator (when PO is approved/denied)

Message Formats:
- PO created: "{creator_name} has created a purchase order: {po_number}"
- PO approved: "Purchase order {po_number} has been approved by {approver_name}"
- PO denied: "Purchase order {po_number} has been denied by {approver_name}. Reason: {reason}"

================================================================================
SPECIAL CASES:
================================================================================

1. BULK OPERATIONS:
   - If admin creates multiple users at once, send one notification per user
   - If stock is allocated to multiple recipients, send individual notifications

2. CASCADE NOTIFICATIONS:
   - When super_admin allocates to manager, notify:
     * manager (recipient)
     * super_admin (confirmation)
     * admin (system-wide event)
   
   - When admin allocates to team_leader, notify:
     * team_leader (recipient)
     * admin (confirmation)
     * super_admin (system-wide event)

3. PRIVACY:
   - Users should only see notifications intended for them
   - super_admin and admin can see system-wide notifications
   - Team leaders see notifications for their team members
   - Agents only see their own notifications

4. NOTIFICATION RETENTION:
   - Keep notifications for 90 days
   - Archive read notifications older than 30 days
   - Allow users to manually delete notifications

================================================================================
PRIORITY LEVELS:
================================================================================

HIGH PRIORITY (Immediate notification):
- Order approvals/rejections
- Stock allocations
- Inventory request approvals/denials

MEDIUM PRIORITY (Standard notification):
- New orders created
- Inventory requests created
- User creation

LOW PRIORITY (Informational):
- System updates
- Team member changes
- Inventory low warnings
